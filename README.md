# 📈 End-to-End Sentiment-Driven Quantitative Trading Pipeline

Welcome to the central repository for the End-to-End Sentiment-Driven Quantitative Trading Pipeline. This project represents an end-to-end quantitative trading infrastructure that fuses real-time market data, deep Natural Language Processing (NLP) for sentiment analysis, and deep learning to predict short-term market movements.

As a modular, enterprise-grade architecture, this pipeline is split into two primary phases: Model Building (Historical ETL & Training) and Live Trading (Real-Time Inference & Dashboarding).

Below is the technical documentation detailing the functionality of each component within the architecture.

<br/>

## 🛠️ Phase 1: Model Building (Historical Data Pipeline)
This phase is dedicated to historical data ingestion, feature engineering, and data quality assurance to create a highly robust dataset for training predictive models (like LSTMs).

<br/>

### Code 1: Deep Insight Historical News Scraper

Functionality: Acts as the primary historical text ingestion engine. It iterates through a predefined list of top Indian stocks and queries historical news using GoogleNews.

NLP & Sentiment: It utilizes newspaper3k to parse full article bodies and NLTK for sentence tokenization. It then feeds the text into a GPU-accelerated ProsusAI/finbert model to extract a "Deep Insight" (the most impactful sentence) and calculates a precise sentiment score.

Factor Mapping: Categorizes the news into macroeconomic, policy, corporate, and financial factors based on keyword matching, outputting a structured CSV.

<br/>

### Code 2: Data Quality & Paywall Repair Script

Functionality: A critical Data Quality Assurance (QA) script designed to clean the historical dataset.

Fallback Logic: Web scrapers inevitably hit paywalls or extract "junk" phrases (e.g., "Please log in to read"). This script identifies these failed rows and re-queries the dates. If a full article scrape fails again, it employs a fallback strategy to synthesize an insight using Google Search snippets, ensuring the final dataset has no missing dates or gaps in sentiment.

<br/>

### Code 3: Feature Engineering & Dataset Unification

Functionality: The core ETL merger. It pulls historical hourly stock data via yfinance with built-in retry logic and robust ticker alias handling.

Technical Indicators: Computes stable, pure-Pandas technical indicators (SMA, RSI, MACD, Bollinger Bands, VWAP) to avoid C++ dependency failures in MLOps environments.

Exponential Decay: Merges the price data with the news sentiment data and applies a crucial 15% hourly exponential decay factor. This mathematically simulates how the market prices in news over time, ensuring the algorithm doesn't trade on "stale" sentiment.

<br/>

### Code 4: Exploratory Data Analysis (EDA.ipynb)

Functionality: Validates the structural integrity of the unified dataset before model training. It applies corporate-styled formatting to generate high-definition visuals.

Business Value: The notebook analytically justifies the feature selection by plotting VWAP against raw close prices. It proves that raw price action is noisy, while SMA provides the 50-hour medium-term trend and VWAP highlights immediate intraday institutional momentum, which are both critical for AI regime mapping.

<br/>

### Code 5: Deep Learning Predictive Engine (LSTM Model)

Functionality: Trains a specialized Long Short-Term Memory (LSTM) neural network for each individual stock ticker to predict the next hour's price movement.

Stationary Engineering: Converts raw, non-stationary stock prices into Log Returns. This mathematically centers the data around zero, allowing the neural network to learn true momentum patterns rather than memorizing infinite price drifts.

Inference Consistency: Applies MinMax scaling (-1 to 1) to normalize diverse features (e.g., millions in Volume vs. decimals in Sentiment). Crucially, it exports these ticker-specific scaler states as .pkl artifacts to ensure the live production environment uses the exact same mathematical boundaries during real-time inference.

Institutional Risk Management: Features a built-in conservative risk engine. The algorithm compares the forecasted price move against its own historical margin of error (RMSE). If the expected profit does not exceed the background market noise (volatility), the AI preserves 
capital by intelligently issuing a "WAIT ⚪" signal, mimicking strict institutional risk-reward protocols.

## ⚡ Phase 2: Live Trading (Real-Time Execution Environment)
This phase handles the live market environment. It is designed for low latency, rate-limit evasion, and real-time decision-making.

<br/>

### Code 1: Stealth Live News Fetcher

Functionality: A continuous daemon process that fetches real-time market news. To evade strict Google rate-limits during live trading, it pivots to the DuckDuckGo Search API (ddgs) using persistent browser sessions.

Sentiment Extraction: Like the historical scraper, it uses FinBERT to extract the most impactful sentence and score its sentiment. It includes a 30-minute exponential backoff protocol if IP soft-blocks are detected, ensuring the bot runs uninterrupted.

<br/>

### Code 2: Live Market Data Broker Connector (Angel One)

Functionality: The direct market bridge. It authenticates with the Angel One SmartApi using API keys and secure TOTP generation.

Dynamic Mapping & Calculation: It downloads the live exchange instrument list, dynamically maps the custom stock aliases to specific broker tokens, and fetches the last 15 days of hourly OHLCV candles. It then calculates the exact technical indicators (RSI, MACD, Volume Shock) required by the trained model using the ta library.

<br/>

### Code 3: Live Data Preprocessor & Merger

Functionality: Prepares the real-time inference payload. It joins the latest live price dataset with the latest stealth news dataset.

Data Formatting: It forward-fills/back-fills news data to cover older hours, applies the identical exponential decay rate used in training, and trims the output to the most recent window (last 15 rows) so the data is perfectly formatted for the model to predict the next hour's price.

<br/>

### Code 4: AI Trading Dashboard & LLM RAG (app.py)

Functionality: The operational command center built on Streamlit. It loads the pre-trained .keras models and scalers, takes the live data payload, and outputs a predicted price and a clear trade signal (BUY 🟢 / SELL 🔴 / WAIT ⚪).

RAG Executive Analyst: Integrates Google's Gemini 2.5 Flash via a Retrieval-Augmented Generation (RAG) architecture. It feeds the live technicals, predicted price, and news headline into the LLM to generate an automated 3-sentence executive verdict. It also features a chatbot for Quants to interrogate the data in natural language.

Automated Journaling: Silently logs every generated prediction, sentiment score, and signal into a local trading_journey.csv file for performance tracking and backtesting.

## 🤖 n8n Workflow Automation Architecture
To ensure the live execution environment runs continuously and reliably, the pipeline utilizes n8n for workflow orchestration and scheduling.

The Master Scheduler: An n8n Cron node triggers the execution loop precisely aligned with live market hourly intervals.

Node 1 (Isolated AI Compute - Colab): The Stealth Live News Fetcher (which runs the heavy FinBERT NLP model) is purposely kept separate from the n8n flow. Due to hardware computation constraints and high processing time, this node is hosted on Google Colab to utilize dedicated GPU instances.

Node 2 (Automated Live Market Data): Fully integrated into n8n. Upon the scheduler's trigger, this node automatically executes the Angel One SmartAPI script to download the latest market candles and compute technical features.

Node 3 (Automated Data Merger): Following Node 2, n8n automatically triggers the Data Preprocessor script, which merges the Colab-generated sentiment CSV with the fresh Angel One technical CSV, seamlessly preparing the final payload for the Streamlit dashboard.

## ⚠️ Critical Engineering Challenges & System Mitigations

1. The Overfitting Dilemma: The LSTM model currently overfits due to data deficiency, requiring Dropout(0.3) layers and an RMSE-based risk engine to halt uncertain trades; however, expanding the historical dataset in the future will ultimately resolve this and achieve long-term model stability.
  
2. Live-Scraping Blockades: To bypass Google's aggressive anti-bot IP bans during live execution, the pipeline pivoted to the DuckDuckGo Search API (ddgs) using persistent browser sessions and randomized exponential backoffs.

3. The Stationarity Problem: To prevent the neural network from failing on infinitely drifting raw stock prices, we engineered the data into stationary Logarithmic Returns, allowing the AI to successfully learn pure market momentum.
